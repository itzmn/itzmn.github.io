java集合-LinkedHashMap

分析LinkedHashMap源码实现，使用场景，以及一些面试问题

简介

LinkedHashMap是在HashMap的基础上实现的，继承了HashMap，在这个基础上扩展，为什么需要这个集合呢，我们在使用HashMap的时候，添加数据他存储是根据key存储，但是不能按照存储顺序得到值，所以就出现了这个集合，可以按照存储顺序得到，还可以根据使用次数得到，底层在HashMap的基础上多了个双向链表

  2. afterNodeAccess(e); 当这个节点是重复节点时，会覆盖原来的节点,然后调用这个方法

我在这整理下添加的具体思路：

  1. 维护顺序是按照添加顺序

    添加值的时候，会新建节点，这个新建节点被LinkedHashMap覆盖，新建了LinkedHashMap的节点，然后调用LinkedLast方法，将这个新加入的节点放进被维护的双向链表的最后端，但是如果添加了相同key的节点，会对节点进行覆盖，然后会调用afterNodeAccess方法，这个方法是在设置维护顺序是按照访问顺序才生效的

  2. 维护顺序是按照访问顺序

    添加值的时候和上面是一样的，但是如果添加了相同key的节点，会对节点进行覆盖，然后会调用afterNodeAccess方法，这个方法是在设置维护顺序是按照访问顺序才生效的，将新覆盖的节点，移动到最后方

总结一下查看

如果集合是按照添加顺序维护的，那么没什么，访问顺序和添加顺序一样

如果是按照访问顺序维护的，那么访问之后，会将刚刚访问的数据维护到双链表的尾部

总结

  1. LinkedHashMap和HashMap有什么联系，区别

    LinkedHashMap是基于HashMap实现的，但是多了双向链表用于维护节点的访问顺序，分为两种，一种是按照添加顺序维护，一种是按照访问顺序维护，遍历较慢

  2. 在迭代的时候能够修改集合么

    不能，因为在迭代的时候会判断集合的修改次数，检验数据的安全性，如果修改会抛出异常

  3. 在LinkedHashMap中数据是有序的，但是转成json很可能就是无序的，

  4. 如何使用这个集合做一个少用即淘汰的策略

    这个集合中有一个删除集合中最老元素的方法，不过其判断条件都是false，不会执行，如果需要可以重写，自定义一下，实现这样的策略
