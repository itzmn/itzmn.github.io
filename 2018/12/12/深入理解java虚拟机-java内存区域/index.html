深入理解java虚拟机-java内存区域

发表于 2018-12-12 | 分类于 java
字数统计: 3.6k 字 | 阅读时长 ≈ 12 分钟
在使用java时候，使用者不用关心内部的内存管理问题，我们看一下数据在java虚拟机内部是如何存储的

java运行时数据区域

java虚拟机在运行程序的时候，会把内存分为几个部分，不同的数据放在不同的位置，先看一下这几个位置

我们下面具体讲一下每个区域

程序计数器

程序计数器是一个较小的内存区域，是当前线程所执行的子节码的行号。java虚拟机多线程处理事情，是通过线程轮流切换实现的，在某一时刻，只会有一条线程在执行，所以为了线程切换之后能够回到刚刚执行的位置，所以就用这个记录刚刚执行的位置，这个是每个线程私有的，

如果线程执行的是一个java方法，这个计数器记录的是正在执行的代码的行号，如果执行的是Native(本地)方法，则存储的值为空，此处区域是虚拟机中唯一一处不会发生内存溢出异常的区域(OutOfMemoryError)

java虚拟机栈

与计数器一样，java虚拟机栈是线程私有的，生命周期有线程相同，虚拟机栈描述的是java方法的内存模型，在执行java方法的时候，就会生成一个栈帧，栈帧压入虚拟机栈，栈帧内部存储，局部变量表，操作数栈，动态链接，方法出口，当方法执行完毕，将栈帧弹出

​ 局部变量表存放了编译期可知的各种基本数据类型（boolean，int，char，short，double，float，long），对象引用(reference类型，他不是对象本身，可能是一个指向对象位置的地址，也可能是一个指向该对象的句柄)，returnAddress(指向了一条字节码指令地址)

​ 其中64位长度的long和double类型的数据会占用2个局部变量空间，其余类型都只是占用1个。局部变量表所需要的内存空间是在编译期完成分配的，当进入一个方法，这个方法所需要在栈帧内分配多大的局部变量表的空间是确定的，不会改变。

​ 栈中会发生两种异常，一种是线程请求的栈的深度大于栈的深度会抛出栈溢出异常(StackOverflowError)，另一种是栈在扩展的时候，无法申请到足够的内存，就会抛出内存异常异常(OutOfMemoryError)

本地方法栈

这个和虚拟机栈很类似，一般我们都合并在一起说栈，区别在于虚拟机栈存储java方法，而本地栈存储本地(Native)方法，本地方法栈也会抛出以上两种异常

java堆

java堆（Heap）应该是虚拟机中内存最大的一块，java堆是被线程共享的，在虚拟机启动时创建，这个地方用来存储对象，几乎所有的对象实例都是在这个地方分配内存。数组也在堆上分配内存，

java堆是垃圾收集管理的主要地域，被称为GC堆。从内存回收的角度看，由于垃圾收集器采用分代收集法，java堆被细分为老年代和新生代（Eden空间8，From Survivor空间1，To Survivor空间1）。

java堆可以是物理上的内存不连续的空间，只要逻辑上的内存是连续的，内存也可以手动设置。

1.7之后，将字符串常量池添加到堆中

方法区

方法区域java堆一样是各个线程共享的。用于存储虚拟机加载的类信息，常量，静态变量，即时编译器编译后的代码。java虚拟机规范将方法区描述为堆的一个逻辑部分，但是他有个别名叫做非堆。

在HotSpot虚拟机上开发，很多人习惯将方法区称作永久代（Permanent Generation），本质上两者不相同，仅仅是HotSpot虚拟机的设计团队将GC分代收集扩展至方法区，或者说使用永久代来实现方法区，这样就可以对这一部分进行内存管理。永久代也是会发生内存溢出的异常的。jdk1.7，已将放在永久代的字符串常量池移出。移至了元空间

这个区域的内存回收主要针对常量池的回收和对类型的卸载

运行时常量池

运行时常量池是方法区的一部分，Class文件中除了有类的版本，字段，方法，接口等描述信息之外，还有一项是常量池，用于存放编译期生成的各种字面量和符号引用，这部分内容在将类加载后进入方法区的运行时常量池中存放。运行时常量池是动态的，可以在运行时候添加数据。

直接内存

这个并不是虚拟机运行时数据区的一部分，也不是java虚拟机规范中定义的内存区域。但这部分内存也被频繁使用，可能导致内存溢出

在jdk1.4添加了NIO类，引入了一种基于通道与缓冲区的io方式，他可以使用native函数分配堆之外的内存，然后通过一个存储在java堆中的DirectByteBuffer对象作为这块内存的引用，进行操作，这样能够提升性能，避免了在java堆中和native堆中来回复制数据。

分配堆外内存，不受jvm限制，但是受本机内存限制，可能会抛出异常

java对象分析

对象创建

我们在创建对象的时候使用了new关键字，看一下具体过程

​ 1.当检测到new指令的时候，会先去常量池中查找是否有这个类的引用，并检查这个引用代表的类是否被加载，解析和初始化过，如果没有执行类加载过程（后面讲）

​ 2.如果已经加载过，为对象分配内存。对象所需要的内存在加载之后就会确定，为对象分配内存是将从java堆中划出一块内存。如果堆内存是很规整的，那么中间回放着一个指针用作分界点，我们只需将指针移动需要的内存大小，这种方法叫做指针碰撞，如果内存不是规整的，那么虚拟机就会维护一个表，上面标注哪一块内存是可用的，我们在列表中找一块内存分配，这种叫做空闲列表。选择哪一种分配方式取决于堆内存是否规整，是否规整取决于垃圾收集器的种类，在使用Serial、ParNew等带整理内存的收集器的时候，会采用指针碰撞，使用CMS这种基于Mark-Sweep算法的收集器使用空闲指针（收集器在后面说）

​ 3.在创建对象时候，修改指针分配内存，这种在多线程容易出现线程安全问题，比如给A分配内存但是线程b也分配了那块内存，就发送问题了，一种是在分配的时候进行同步处理，虚拟机采用这种，Cas配上失败重试的方式保证更新操作的原子性，另一种是把内存按照线程分为不同的区域，每个线程在java堆中有一个自己的缓存内存（ThreadLocal），分配内存在自己的内存中分配，不够就再次申请。在申请的时候需要进行同步

​ 4.内存分配完毕之后，将分配到的内存空间初始化为零值（不包括对象头），这一步保证对象的实例字段在java代码中不赋值初始值就可以直接使用，返回对应数据类型的0值。

​ 5.接下来，虚拟机对对象进行必要的设置，例如这个对象是哪个类的实例，如何才能找到类的元数据信息，对象的hash吗，对象的GC分代年龄等。这些信息放在对象头中。

上面做完，在虚拟机看来，对象已经产生，对于java程序来说，还需要执行init方法，所有字段都是零，

对象的内存分布

对象在内存中的存储布局分为3块区域，对象头，实例数据，对象填充。

1.对象头包括两部分信息，一部分用于存储对象自身的运行时数据，hashCode，GC分代年龄，锁状态，线程持有的锁，偏向线程id，偏向时间戳等，这部分数据在32位和64位虚拟机中粉笔为32bit和64bit，官方称之为“Mark Word”。对象需要存储的数据很多，其实已经超过了32位64位，但是对象头信息是与自身定义的数据无关的额外存储成本，为了尽可能存储多数据，会根据对象的状态复用自己的存储空间，如在32位虚拟机，对象未被锁定，25bit存储hashCode，4bit存储对象的gc年龄，2位存储锁标志位，1位固定为0，其他状态存储内容，如下

对象头的另一部分是类型指针，即对象指向他的类元数据的指针，虚拟机通过这个指针来确定这个对象是哪个类的实例。如果对象是一个数组，那在对象头中还存储数组长度的数据

2.实例部分是对象真正存储的有效信息，也是在程序代码中定义的各种类型字段内容。这部分的存储顺序按照字段宽度分配。

3.第三部分的对齐填充并不是必然存在的，起着占位符的作用，虚拟机要求对象的起始地址必须是8个字节，换句话说对象的大小必须是8字节的倍数，对象头正好是8的倍数，实例数据部分没有对齐，就由这部分补充。

对象的访问定位

建立对象是为了使用对象，我们的java程序需要通过栈上的reference数据来操作堆上的具体对象。由于reference类型在虚拟机规范中是一个指向对象的引用，并没有定义如何通过引用去定位，

1。句柄，java堆中会划分出句柄池，reference中存储的就是句柄的地址，句柄中包含了对象实例数据与类型数据各自的具体地址信息

2.直接指针访问，java堆对象的布局中必须考虑如何放置访问类型数据相关信息，而reference中存储对象的地址

使用句柄访问的好处就是reference中存储的是句柄地址，当对象移动的时候，会改变句柄中的对象地址，而reference不用改变。

使用指针访问，速度快，不用再通过一次句柄去访问，虚拟机HotSpot默认就是这种

实战java内存异常

参数设置

命令 含义
-Xms 堆的初始内存
-Xmx 最大堆大小。如：-Xmx512m
-Xmn 新生代大小。通常为 Xmx 的 1/3 或 1/4。新生代 = Eden + 2 个 Survivor 空间。实际可用空间为 = Eden + 1 个 Survivor，即 90%
-Xss JDK1.5+ 每个线程堆栈大小为 1M，一般来说如果栈不是很深的话， 1M 是绝对够用了的。
-XX:NewRatio 新生代与老年代的比例，如 –XX:NewRatio=2，则新生代占整个堆空间的1/3，老年代占2/3
-XX:SurvivorRatio 新生代中 Eden 与 Survivor 的比值。默认值为 8。即 Eden 占新生代空间的 8/10，另外两个 Survivor 各占 1/10
-XX:PermSize 永久代(方法区)的初始大小
-XX:MaxPermSize 永久代(方法区)的最大值
-XX:+PrintGCDetails 打印 GC 信息
-XX:+HeapDumpOnOutOfMemoryError 让虚拟机在发生内存溢出时 Dump 出当前的内存堆转储快照，以便分析用
-XX:+PrintGCTimeStamps 打印此次垃圾回收距离jvm开始运行的所耗时间。
-Xloggc: 将垃圾回收信息输出到指定文件
-verbose:gc 打印 GC 日志
-XX:+PrintTenuringDistribution 对象晋升的日志
-XX:+PrintGCApplicationStopedTime 查看 gc 造成的应用暂停时间

异常

堆溢出

前面说了，堆存储的是对象实例，所以只要够多的对象实例，就会报异常，

异常

堆溢出，分为两种可能，一种是内存泄漏，一种是内存溢出，内存泄漏是站着茅坑不拉屎，站着位置，啥事不干，内存溢出是对象都有用，但是对象过多，我们先看看快照，是否是内存泄漏，某些对象是否是相互引用，而不是在GC Root的引用链上，那么就是内存泄漏，代码有问题，如果是内存溢出，那就要修改堆内存的大小

栈异常

栈异常分为 StackOverflowError，还有OutOfMemoryError两种，当递归的深度太深，抵达了栈的最大深度，则会报出栈溢出异常，当栈扩容时申请不到内存则抛出内存溢出异常

方法区异常

OOM：PermGen space

方法区存储类信息，即时编译的类数据，当动态生成大量的类，会撑爆方法区

总结

  1. 在jdk1.7之后，将字符串常量池移动到了堆中
  2. 方法区也被称为永久代，但是1.8之后被元空间取代
