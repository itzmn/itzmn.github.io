深入理解java虚拟机-垃圾收集器

发表于 2018-12-12 | 分类于 java
字数统计: 4.4k 字 | 阅读时长 ≈ 15 分钟
了解java内部是如何管理内存的，如何清除无用的数据的

垃圾收集分为三个部分，收集那些数据，什么时候回收，怎么回收

对象已死吗

垃圾收集器收集对象的时候，首先要确定这个对象是否无用，然后在确定是否回收

引用计数法

给对象添加一个引用计数器，每当有一个地方引用到它时，计数器的值就加1，当引用失效时候，计数器值就减1，任何时刻计数器值为0的对象就是不可能在被使用的。

引用计数器方法实现简单，效率也很高，但是有个问题，就是当两个无用对象互相引用时候，这样就会造成内存浪费。

可达性分析

主流的程序语言（java）使用的就是可达性分析来判定对象是否存活的。

通过一系列的GC Root的对象作为起点，从这些节点开始从上到下寻找，搜索走过的路线叫做引用链，当一个对象没有在引用链上，说明这个对象不可用，是可以回收的

可以作为GC Root的对象：

  • 虚拟机栈（栈帧中的局部变量表）中引用的对象
  • 方法区中类静态属性引用的对象
  • 方法区中常量引用的对象
  • 本地方法栈中JNI（Native方法）引用的对象

什么是引用

引用，在jdk1.2以前，如果reference类型的数据中存储的数值代表的是另外一块内存的起始地址，就称这块内存代表着一个引用。这样对象就只有两种状态，被引用，没有被引用。

在jdk1.2以后，出现了扩充，出现了强引用(Strong Reference)，软引用(Soft Reference)，弱引用(Weak Reference)，虚引用(Phantom Reference)。引用强度逐渐减弱

  • 强引用，在程序代码中普遍存在的，类似“X x = new X()”这类引用，只要强引用还在，垃圾收集器永远不会回收掉被引用的对象
  • 软引用，用来描述一些还有用但是并非必须的对象，对于软引用关联着的对象，在系统将要发生内存溢出异常的时候，将会把这些对象进行回收，如果还是没收到足够的内存，抛出异常
  • 弱引用，也是用来描述为必须对象的，但是它的强度比软引用要差一点，被弱引用的对象只能生活到下一次垃圾收集发生之前，当垃圾收集器工作时，无论当前内存是否足够，都会回收掉被弱引用的对象。
  • 虚引用被称为幽灵引用，他是最弱的引用。一个对象是否有虚引用的存在，完全不会对其生存时间构成影响，也无法通过虚引用来取得一个对象实例。为一个对象设置虚引用的唯一目的就是能在这个对象被回收时收到一个系统通知。

生存还是死亡

即使在可达性分析算法中不可达的对象，也不是非死不可，这时候他们处于缓刑状态，要真正宣告一个对象死亡，至少需要经历两次标记过程，如果对象在进行可达性分析后没有发现与GC ROOT 相连接的引用链，那它将会被第一次标记并且进行一次筛选，筛选的条件是此对象是否有必要执行finalize()方法(Object内部方法)，当对象没有覆盖该方法时，或者finalize方法已经被虚拟机调用过，虚拟机将这两种情况视为没有必要执行

如果这个对象被判定有必要执行finalize方法，那么将这个对象放置在一个叫F-Queue的队列之中，并在稍后由一个由虚拟机自动建立的，低优先级的Finalize线程去执行 它。这里所谓的执行就是虚拟机会调用这个方法，但并不承诺会等待它运行结束，这样做的原因是，如果一个对象在finalize中执行缓慢，或者死循环，很可能会导致队列中其他对象永久等待。finalize方法是对象逃脱死亡的最后一次机会，GC会对F-Queue中的对象进行筛选，如果在执行过程中重新连接上引用链，那么就活了，不然就直接回收了。

回收方法区

方法区，也被称为永久代，收集主要分为两个部分：废弃常量和无用的类。回收废弃常量与回收java堆中的对象非常相似，以常量池中字面量的回收为例，假如一个字符串“abc”已经进入了常量池，当时当前系统没有一个String对象叫做“abc”，没有String对象引用这个常量，如果这个时候发生内存回收，这个常量就会被回收掉

回收类。

判断类是否是无用的类

  • 该类所有实例都被回收
  • 加载该类的类加载器已经被回收
  • 该类对应的java.lang.Class对象没有在任何地方被引用，无法在任何地方通过反射访问该类的方法

垃圾收集算法

标记-清除算法

最基础的收集算法就是标记清除算法，分为标记和清除两个阶段。首先标记出所有需要回收的对象。在标记完成之后统一回收被标记的对象，它的标记在上面已经讲过了。

不足：

  • 效率不高，标记和清除两个过程效率都不高
  • 空间问题，标记清除之后会产生大量的不连续的内存碎片，空间碎片太多可能会导致后面需要分配大内存的时候找不到连续内存产生下一次GC。

复制算法

为了解决效率问题，将内存划分为同大的两块，每次只使用其中一块，当内存用完，触发gc，将存活的对象复制到另一块内存中按顺序摆放，把已使用的内存清空，这样每次都是对半个区回收，内存分配也很好，通过指针碰撞分配。不用考虑空间碎片

不足：

  • 内存缩小为原来一般

现在商业虚拟机都是使用这种算法回收新生代，因为新生代存活时间较短，将内存分为3个部分一块较大的Eden和两块较小的Survivor内存，每次使用Eden和一个Survivor。当回收时候将Survivor和Eden中的存活数据拷贝到另一个Survivor中，清理掉以前使用的，默认Eden和Survivor大小比例是8:1，当Survivor内存不够用的时候，就会将数据放在老年代中去。

标记整理算法

复制算法当存活对象过多的时候，需要进行大量的复制工作，效率较低，而且会浪费一半的内存。

老年代主要使用标记整理算法，标记整理过程和标记清除一样，后序步骤不是直接对可回收对象进行整理，而是让所有活着的对象移向一端，清理掉剩下的空间。

分代收集算法

把java堆分为新生代和老年代，根据每个年代的对象的特点使用不同的收集算法。新生代存活率比较低，使用复制算法，老年代存活较多，使用标记清理或者标记整理算法。

垃圾收集器

在虚拟机中可能会使用很多种垃圾收集器，我们看一下有哪些

如果两个收集器中间有线，说明可以组合使用，垃圾收集器所在区域，就是那个区域的收集器

Serial收集器

Serial收集器是最基本、发展历史最悠久的收集器，在jdk1.3之前新生代唯一的选择。这是一个单线程收集器使用复制算法。在进行垃圾收集时，必须暂停其他所有的工作线程知道收集结束。“Stop the World”，这项工作是由虚拟机在后台自动发起执行完成的。在用户不可见的情况下吧用户正常工作的线程全部停掉。

Serial收集器是虚拟机运行在Client模式下默认的新生代收集器。（一般安装的都是Client模式）

Stop the world ： 当垃圾收集器在收集清除垃圾时候，如果不停止，还会产生，比较不好，所以要停止

ParNew收集器

ParNew是Serial的多线程版本，除了使用多条线程进行垃圾收集之外，其余行为包括Serial收集器可用的所有控制参数，收集算法，Stop the World，对象分配原则，回收策略都和Serial收集器一样。

这个收集器是Server模式的虚拟机的首选的新生代收集器，因为只有它和Serial收集器可以与CMS收集器（老年代收集器）配合使用。

Parallel Scavenge收集器

Parallel Scavenge收集器是一个新生代收集器，他也是使用的复制算法。

特点是与别的收集器关注点不同，别的收集器关注点是缩短GC停顿时间，而Parallel Scavenge收集器的目的则是达到一个可以控制的吞吐量。所谓吞吐量就是cpu用于运行用户代码的时间与CPU总消耗时间的比值，吞吐量=运行用户代码时间/ （用户代码时间+垃圾收集时间）

停顿时间越短就越适合需要与用户交互的程序，良好的响应速度能提升用户体验，而高吞吐量则可以更高效的利用cpu时间，尽快完成程序的运算，主要适合在后台运算而不需要太多交互任务

Parallel Scavenge收集器提供了两个参数用于精确控制吞吐量，分别是控制最大垃圾收集停顿时间的-XX:MaxGCPauseMillis，和直接设置吞吐量大小的-XX:GCTimeRatio参数，MaxGCPauseMillis是一个大于0的毫秒数，GC停顿时间是以牺牲吞吐量和新生代空间换来的，系统会把新生代调小，收集会快，但是收集的次数也会变多。GCTimeRatio参数的值是一个大于0切小于100的整数，也是垃圾收集时间占时间的比率，相当于吞吐量的倒数。

收集器会自动调整这些参数，达到最合适的停顿时间或者最大的吞吐量，这叫做自动调节策略。是ParNew没有的。

Serial Old收集器

Serial Old收集器是Serial的老年代版本收集器，同样是一个单线程收集器，使用标记整理算法。收集器主要是Client模式的虚拟机使用。如果是server模式，一种用途是在1.5之前与Parallel Scavenge收集器搭配使用，一种是座位CMS的后备预案。

Parallel Old 收集器

Parallel Old是Parallel Scavenge收集器的老年代版本。使用多线程和标记整理算法，是1.6之后才有的。主要是为了和Parallel Scavenge配合使用，使得达到高的吞吐量。

两个收集器配合使用

CMS收集器

CMS收集器是一种以获取最短回收停顿时间为目标的收集器。目前一些B/S系统的网站，希望停顿时间最小，交互感最好。

CMS使用标记清除算法。分为四个步骤

  • 初始标记
  • 并发标记
  • 重新标记
  • 并发清除

初始标记与重新标记需要stop the world 。初始标记仅仅是标记一下GC root能直接关联到的对象，速度很快。并发标记阶段是进行GC Roots Tracing的过程，而重新标记则是为了修正并发标记期间因为用户程序继续运作导致标记产生变动的那一部分对象，这个阶段停止会比初始标记阶段长一点，但远比并发标记时间短。

由于整个过程中耗时最长的并发标记，和并发清除过程都可以与用户线程一起工作，所以，从总体来看，CMS收集器的内存回收过程是与用户线程一起并发执行的。

缺点：

  • 会占用一些cpu，在并发程序中，会分出一部分cpu资源给收集器线程。与用户线程一起执行，会导致应用程序变慢，总吞吐量降低。
  • CMS无法处理浮动垃圾，当cms在并发标记的时候是与用户线程同时执行的，标记不到用户线程新产生的垃圾，只能等待下一次gc，这就是浮动垃圾。在垃圾收集过程中用户还在运行，所以需要预留出一部分内存给用户线程产生新的垃圾，1.5默认使用老年代68%，可以使用-XX:CMSInitiatingOccupancyFraction的值来修改。1.6中改成了92%，在CMS运行过程中，新产生的垃圾如果剩下的空间不够用，就会出现“Concurrent Mode Failure”失败，这时虚拟机启动后备预案，使用Serial Old，进行老年代的垃圾收集，这样停顿时间就长了。
  • 标记清除容易产生大量的空间碎片。

G1 收集器

G1是一款面向服务端的收集器,G1将整个堆划分为多个大小相同的独立区域，保留新生代和老年代的概念。但新生代和老年代不再是物理隔离的，是一部分独立区域的集合。G1跟踪每个区域，维护里面的垃圾堆积的价值大小（回收所获得空间与收集时间的比值）表，根据允许的收集时间回收价值最大的区域

特点

  • 并行与并发： G1能够利用多cpu，多核环境缩短Stop the World的时间。部分其他收集器原本需要停顿的动作，G1可以通过并发的方式让java程序继续执行
  • 分代收集：G1可以单独管理整个堆，但是还是可以使用不同的方式去处理不同的对象。
  • 空间整合，G1整体看是使用标记整理算法，局部看是使用复制算法，不会产生碎片。
  • 可预测的停顿，G1除了追求停顿，还能建立可预测的停顿时间模型。能够让使用者明确指定在一个长度为M毫秒的时间片段，消耗在垃圾收集上的时间不得超过N毫秒。

虚拟机使用Remembered Set避免全堆扫描，每次对Reference类型数据进行写操作，会检查Reference引用的对象是否处于不同的区域，如果是变将相关的引用信息记录到被引用对象所属的区域的Remembered Set中，当进行回收，就可查询。

内存分配与回收策略

对象优先在Eden分配

大多数情况下，对象在新生代Eden区进行分配，当Eden没有足够空间进行分配时，虚拟机执行Minor GC（新生代GC）。

大对象直接进入老年代

所谓的大对象指的是，需要大量连续内存空间的java对象，最典型的就是很长的字符串，与数组。

虚拟机提供了一个-XX:PretenureSizeTHreshold参数，令大于这个值的对象直接进入老年代，

长期存活的对象进入老年代

每个对象都有一个GC年龄，这个存储在对象头中，前面讲过了。如果对象在eden出生经过一次Minor Gc 仍然能活，移动到另一个Survivor，对象设为1岁，每熬过一次，增加1岁，当增加到一定程度（默认15岁）就会进入老年代。可以通过-XX:MaxTenuringThreshold设置

动态年龄判定

当Survivor中相同年龄所有对象大小的总和大于Survivor的一半，年龄大于等于该年龄的对象可以直接进入老年代。

空间分配担保

在发生Minor GC之前，虚拟机会先检查老年代最大可用连续空间是否大于新生代所有对象总空间，如果大于，说明这次gc是安全的。如果不成立，虚拟机会查看HandlePromotionFailure设置值是否允许担保失败，如果允许，那么会检查老年代最大可用连续空间是否大于历次晋升到老年代对象的平均大小，如果大于，则进行Minor GC,如果小于，或者设置为不担保失败进行Full GC。
