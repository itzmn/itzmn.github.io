并发编程-java并发机制的底层实现原理

发表于 2018-12-14 | 分类于 java ， 并发编程
字数统计: 3.5k 字 | 阅读时长 ≈ 11 分钟
java代码在编译后会变成java字节码，字节码被类加载器加载到jvm里，jvm执行字节码，最终需要转化成汇编指令在CPU执行，java中所使用的并发机制依赖于jvm的实现和CPU指令，我们看一下底层实现机制

volatile的应用

多线程并发编程中synchronized和volatile都扮演着重要的角色，volatile是轻量级的synchronized，他在多处理器开发中保证了共享变量的可见性。可见性的意思是当一个线程修改一个共享变量时，另外一个线程能读到这个修改的值。如果volatile变量修饰符使用恰当的话，他比synchronized的使用成本低，因为他不会引起线程上下文的切换和调度。

volatile的定义与实现原理

java编程语言允许线程访问共享变量，为了确保共享变量能被准确和一致地更新，线程应该确保通过排它锁单独获得这个变量。java提供了volatile，在某些情况下比锁要方便。如果一个字段被声明为volatile，java线程内存模型确保所有线程看到这个变量的值是一致的。

先看一个CPU的术语定义

术语 英文 描述
内存屏障 memory barriers 是一组处理器指令，用于实现对内存操作的顺序限制
缓冲行 cache line 缓存中可以分配的最小存储单位。处理器填写缓存线时会加载整个缓存线，需要使用多个主内存读周期
原子操作 atomic operations 不可中断的一个或一系列操作
缓存行填充 cache line fill 当处理器识别到内存中读取操作数是可缓存的，处理器读取整个缓存行到适当的缓存
缓存命中 cache hit 如果进行高速缓存行填充操作的内存位置仍然是下次处理器访问地址时，处理器从缓存中读取操作数，而不是从内存中读取
写命中 write hit 当处理器将操作数写回到一个内存缓存的区域时，它首先会检查这个缓存的内存地址是否在缓存行中，如果存在一个有效的缓存行，则处理器将这个操作数写回到缓存，而不是协会到内存，这个操作被称为写命中
写缺失 write misses the cache 一个有效的缓存行被写入到不存在的内存区域

volatile是如何保证可见性的呢？

代码最终生成的汇编代码中，有volatile变量修饰的共享变量进行写操作的时候会多出一行汇编代码，作用

1.将当前处理器缓存行的数据写回到系统内存

2.这个写回内存的操作会使其他CPU里缓存了该地址的数据无效

为了提高处理速度，处理器不直接和内存进行通信，而是将系统内存中的数据读取到内部缓存，但操作完不知道何时会写到内存。如果对声明了volatile的变量进行写操作，jvm就会向处理器发送一条Lock前缀的指令，将这个变量所在的缓存行的数据写回到内存中。但是就算写回到内存，如果其他处理器缓存的值还是旧的，再执行计算操作就会有问题。所以，在多处理器下，为了保证各个处理器的缓存是一致的，就会实现缓存一致性，每个处理器通过嗅探在总线上传播的数据来检查自己的缓存是否过期了，如果自己缓存行对应的内存地址被修改，就会将当前 处理器的缓存行设置为无效，当处理器对这个数据进行操作修改的时候，会重新从系统内存中把数据读到处理器缓存中。

  1. Lock前缀指令会引起处理器缓存写回内存

    在最近的处理器中，Lock信息一般不锁总线，而是锁缓存， 会锁定这块内存区域的缓存，并将缓存写回内存，并使用缓存一致性机制，来确保修改的原子性，此操作被称为缓存锁定，缓存一致性会阻止同时修改由两个以上处理器缓存的内存区域数据

  2. 一个处理器的缓存回写到内存，使得其他处理器的缓存无效

    处理器使用嗅探技术保证它的内部缓存，系统内存和其他处理器缓存的数据在总线保持一致，当通过嗅探一个处理器来监测其他处理器打算写内存地址，而这个地址当前处于共享状态，那么正在嗅探的处理器将使它的缓存行无效，在下次访问地址时，强制执行缓存行填充。

synchronized的实现原理

synchronized以前被称为重量级锁。但是随着1.6对他的优化，引入了偏向锁和轻量级锁

java中的每一个对象都可以作为锁

  • 对于普通同步方法，锁的是当前实例对象
  • 对于静态同步方法，锁的是当前类的Class对象
  • 对于同步方法块，锁的是synchronized括号里面配置的对象

当一个线程试图访问同步代码块的时候，他首先必须得到锁，退出或者抛出异常的时候必须释放锁。

synchronized实现原理（基于监视器），jvm基于进入和退出Monitor对象来实现方法同步和代码块同步的，代码块同步是使用monitorenter和monitorexit实现的。

monitorenter插在同步代码块的开始位置，而monitorexit是插入到方法结束处和异常处，jvm保证每个monitorenter必须有对应的monitorexit与之配对。任何对象都有一个monitor与之关联，当且一个monitor被持有后，他将处于锁定状态，直到解锁

java对象头

synchronized用的锁是存在于java对象头中的。如果java对象是数组类型，虚拟机用3个字宽存储对象头，如果是非数组类型，则用2个字宽存储对象头。

锁升级

jdk1.6为了减少获得锁和释放锁带来的性能消耗，引入了偏向锁和轻量级锁，在java se 1.6中，锁一共有4中状态，分别是无锁，偏向锁，轻量级锁，重量级锁。这几个状态会随着竞争情况逐渐升级。锁可以升级但是不能降级。这种能升不能降是为了提高获得锁和释放锁的效率

偏向锁

大多数情况下，锁不仅不存在多线程竞争，而且总是由同一线程多次获得，为了让线程获得锁的代价更低而引入偏向锁。当一个线程访问同步块并获得锁之后，会在java对象头和栈帧中的锁记录中存储锁偏向的线程id，以后该线程进入和退出该同步块时不需要进行CAS操作来进行加锁和解锁，只需要测试一下对象头的Mark Word里是否存储着指向当前线程的偏向锁。如果测试成功，表示线程已经获得锁。如果测试失败，需要测试一下Mark Word中偏向锁的标识是否设置为1（标识当前是偏向锁），如果没有设置，则CAS竞争锁，如果设置了，则尝试使用CAS将对象头的偏向锁指向当前线程。

1.偏向锁的撤销

偏向锁使用了一种等到竞争出现才释放锁的机制。所以当其他线程尝试竞争偏向锁时，持有偏向锁的线程才会释放锁。偏向锁的撤销，需要等到全局安全点（这个时间没有正在执行的字节码）。他会首先暂停拥有偏向锁的线程，然后检查持有偏向锁的线程是否或者，如果线程不处于活动状态，则将对象头设为无锁，如果线程或者，拥有偏向锁的栈会被执行，遍历偏向对象的锁记录，栈中的锁记录和对象头的Mark Word要么重新偏向于其他线程，要么回复到无锁状态或者标记对象不适合座位偏向锁，最后唤醒暂停的线程。

偏向锁在java6和java7是默认启用的，但是它在应用程序启动几秒之后才激活，如有必要可以使用jvm参数关闭延迟。

轻量级锁

轻量级加锁

线程在执行同步块之前，jvm会先在当前线程的栈帧中创建用于存储锁记录的空间，并将对象头中的mark word复制到锁记录中，然后线程尝试使用cas将对象头中的mark word替换为指向锁记录的指针。如果成功则当前线程获得锁，如果失败，表示其他线程竞争锁，当前线程便尝试 使用自旋来获取锁。

轻量级解锁

轻量级解锁时，会使用原子的CAS操作将拷贝过来的对象头的信息替换回对象头，如果成功，则表示没有竞争发生。如果失败，表示当前锁存在竞争，锁就会膨胀为重量级锁。

因为自旋会消耗CPU，为了避免无用的自旋（比如获得锁的线程被阻塞住了），一旦锁升级为重量级锁，就不会在恢复到轻量级锁状态。当锁处于这个状态下，其他线程试图获取锁时，都会被阻塞住，当持有锁的线程释放锁之后会唤醒这些线程，被唤醒的线程就会进行新一轮的夺锁之争。

锁的优缺点

锁 优点 缺点 适用场景
偏向锁 加锁和解锁不需要额外额消耗，和执行非同步方法相比仅存在纳秒级别的差距 如果线程间存在锁竞争，会带来额外的锁撤销的消耗 适用于一个线程访问同步块场景
轻量级锁 竞争的线程不会阻塞，提高了程序的响应速度 如果始终得不到锁竞争的线程，使用自旋会消耗CPU 追求响应速度，同步块执行速度非常快
重量级锁 线程竞争不使用自旋，不会消耗CPU 线程阻塞，响应时间缓慢 追求吞吐量，同步块执行速度较长

原子操作的实现原理

原子操作是“不可被中断的一个或一系列操作”

CAS

CAS（compare and swap）比较并交换：cas操作需要输入两个数值，一个旧值（期望操作之前的值），和一个新值，在操作期间先比较旧值有没有发生变化，如果没有发生变化，才交换成新值，发生变化则不交换。

处理器如何实现原子性

通过缓存加锁和总线加锁方式

  • 总线加锁

    如果多个处理器同时对共享变量进行读改写操作(i++)，那么共享变量就会被多个处理器同时进行操作，这样读改写操作就不是原子了，操作完成之后共享变量的值会合期望的不一样。

    原因是可能多个处理器同时从各自的缓存中读取变量进行操作，然后写回内存，值就有误差了。

    总线锁，当一个处理器在总线上输出LOCK信号时，其他处理器的请求被阻塞，该处理器就可以独占共享内存了。

  • 缓存锁

    在同一时刻，我们只需要保证对某个内存地址的操作是原子即可，总线锁将锁住所有内存地址，不能操作，比较浪费。

    缓存锁使用缓存锁定实现的，缓存锁定是指内存区域如果被换粗你在处理器的缓存行中，并在lock期间被锁定，当么当他写回内存的时候，修改内存的内存地址，并允许他的缓存一致性机制来保证操作的原子性，因为缓存一致性会阻止同时修改由两个以上的处理器缓存的内存区域数据，当其他处理器回写到已被锁定的缓存行的数据时，缓存行就无效了。

java实现原子性

java中通过锁和循环cas的方式实现原子操作。

使用循环cas实现原子操作

jvm中的cas操作利用了处理器提供的CMPXCHG指令实现的。自旋CAS实现的基本思路就是循环进行cas操作，知道成功为止。

CAS问题

  1. ABA问题

    因为cas需要在操作值的时候，检查值有没有发生变化，如果没有则操作，如果一个值是a,被改成b，又改成了a，那么使用cas会发现这个数没有变。

    解决思路：使用版本号，变量添加版本号

  2. 循环时间长开销大

    自旋CAS长时间不成功，给CPU带来很大的消耗。

  3. 只能保证一个共享变量的原子操作

    当对一个共享变量执行操作时，我们可以使用循环cas来保证原子操作，但是对多个共享变量cas无法保证，需要用锁，

使用锁实现

锁保证只有获得锁的线程才能够操作锁定的内存区域。jvm中实现了很多锁机制，偏向锁，轻量级锁，互斥锁。除了偏向锁，实现方式都使用了CAS，当一个线程进入同步块的时候使用cas获得锁，使用cas释放锁。
